## 摘要
视觉 Transformer (ViT) 的 Token 剪枝方法通过丢弃无信息的补丁，有望实现注意力机制 FLOPs 的平方级减少。然而，当使用最先进的变长注意力 API（包括 FlashAttention-2 的 varlen 和 PyTorch 的 NestedTensor SDPA）执行剪枝后的序列时，实际运行延迟并未相应缩减。我们将此归因于调度开销瓶颈：在 ViT 典型的剪枝后短序列长度（<=197 个 token）下，实际矩阵运算在个位数微秒内即可完成，而主机端调度路径却消耗了 60-90 微秒。我们提出了一种轻量级的双向 Triton 注意力内核，其调度底噪为 40 微秒，比 FlashAttention-2 varlen 低约 1.5 倍，从而使剪枝带来的节省在实际运行时间中更加显著。通过集成到一个完整的“打包-注意力-解包”流水线中，我们的系统在四种剪枝算法（Threshold-L2、DynamicViT、EViT、ATS）上一致实现了相比填充 PyTorch SDPA 高达 2.24 倍的端到端吞吐量，支持 DeiT-T/S/B 规模的扩展，并保持了位精确的分类预测，最大绝对 Logit 差异小于 0.007。

## Abstract
Token pruning methods for Vision Transformers (ViTs) promise quadratic reductions in attention FLOPs by dropping uninformative patches. Yet when pruned sequences are executed with state-of-the-art variable-length attention APIs -- including FlashAttention-2's varlen and PyTorch's NestedTensor SDPA-the wall-clock attention latency doesn't scale accordingly. We trace this to a dispatch-overhead bottleneck: at the short, post-pruning sequence lengths typical of ViTs (<=197 tokens), actual matrix arithmetic completes in single-digit microseconds while the host-side dispatch path consumes 60-90 us. We present a lightweight, bidirectional Triton attention kernel whose dispatch floor is 40 us roughly 1.5x lower than FlashAttention-2 varlen-allowing pruning savings to become more visible in wall-clock time. Integrated into a complete pack-attend-unpack pipeline, our system achieves up to 2.24x end-to-end throughput over padded PyTorch SDPA consistently across four pruning algorithms (Threshold-L2, DynamicViT, EViT, ATS), scales across DeiT-T/S/B, and maintains bit-exact classification predictions with <0.007 max absolute logit difference.

### 1. 论文的核心问题与整体含义
*   **研究动机**：Token 剪枝（Token Pruning）是减少视觉 Transformer (ViT) 计算量（FLOPs）的常用手段。然而，研究者发现理论上的计算量减少往往无法转化为实际的推理加速。
*   **核心问题**：现有的变长注意力 API（如 FlashAttention-2 varlen）主要针对大语言模型（LLM）的长序列设计。在 ViT 场景下，剪枝后的序列极短（通常 $\le 197$ tokens），此时**主机端调度开销（Dispatch Overhead）**（如 Python 参数验证、内核启动等）远超实际的 GPU 计算时间。这导致无论剪枝掉多少 Token，推理延迟都卡在一个“底噪”水平，无法进一步降低。

### 2. 论文提出的方法论
*   **核心思想**：开发一种专门针对 ViT 短序列优化的轻量级 Triton 注意力内核，通过简化调度路径来降低延迟底噪。
*   **关键技术细节**：
    *   **调度优化**：利用 OpenAI Triton 的 JIT 编译器生成精简的内核启动存根（Stub），绕过 `pybind11` 边界和复杂的 C++ 参数封送，将调度底噪从 ~62 $\mu s$ 降至 ~40 $\mu s$。
    *   **不规则注意力内核（Ragged Attention）**：实现了一个专门针对双向（非因果）注意力的内核，支持变长 Batch。它采用 FlashAttention-2 的在线 Softmax 算法，但去除了 LLM 所需的因果掩码、Dropout 和 KV 缓存等复杂功能，以保持极简。
    *   **融合 Token 打包（Fused Packing）**：设计了一个两阶段流水线。首先计算索引偏移，然后使用自定义 Triton 内核将存活的 Token 快速收集到连续的扁平缓冲区中，避免了 PyTorch 原生操作的冗余开销。
    *   **端到端集成**：将该内核集成到 ViT 的推理流程中，使剪枝后的 MLP 层和注意力层都能在紧凑的“不规则”张量上运行。

### 6. 论文的主要结论与发现
*   **调度瓶颈是真实存在的**：在小 Batch 和短序列下，FlashAttention-2 的延迟几乎是恒定的（~63 $\mu s$），无论剪枝比例如何。
*   **性能提升显著**：相比填充后的 PyTorch 基准，该方法实现了高达 **2.24 倍** 的端到端吞吐量提升。

*   **剪枝收益的重新分配**：发现剪枝带来的实际加速主要源于 MLP 层的线性缩放，而注意力层的加速必须通过降低调度开销才能释放。

### 7. 优点
*   **直击痛点**：敏锐地捕捉到了学术界剪枝算法与工业界底层算子实现之间的“脱节”问题。
*   **系统级优化**：不只是改进算法，而是通过重新设计“打包-计算-解包”的整个系统流程来解决问题。
*   **易于集成**：该方法对剪枝算法是不可知的（Agnostic），可以作为现有多种剪枝研究的通用加速插件。

### 8. 不足与局限
*   **硬件依赖性**：虽然在 T4 上有测试，但核心优化高度依赖于 Triton 编译器对特定架构的优化质量。
*   **功能局限**：为了追求极致的低调度开销，该内核舍弃了 Dropout、因果掩码等功能，这意味着它无法直接用于自回归解码（如 GPT 推理）或需要训练的场景。
*   **算子效率上限**：在纯计算性能（指令级优化）上，Triton 生成的代码在极高负载下仍略逊于高度手工优化的 CUDA/CUTLASS 内核。
